# Kotlin until 문법
until은 끝을 포함하지 않는(exclusive) 범위를 만드는 함수

for (i in 0 until from.keyCount)

이건 i를 0부터 from.keyCount - 1까지 반복하라는 뜻,
즉 keyCount는 포함하지 않는다.

다른 범위 문법 비교,

0 until keyCount -> 0 <= i < 3 -> 0,1,2
0..keyCount -> 0 <= i <= 3 -> 0, 1, 2, 3
0..<keyCount -> 0 <= i < 3 ->0, 1, 2

## 배열 리스트 순회에 until을 쓰는 이유
인덱스는 항상 0부터 size - 1까지이기 때문이다. keyCount개의 원소가 있으면 유효한 인덱스는 0 ~ keyCount-1이다.
만약 0..keyCount를 쓰면 마지막에 keyAt(keyCount)를 호출해서 인덱스 범위 초과 (out of bounds) 에러가 난다.

이때 until은 중위 함수(infix function)이며,
```java
0 until from.keyCount
0.until(from.keyCount)
```
위의 2개의 메서드는 같다.

# B-Tree 범위 스캔 알고리즘(range scan)
1. findLeafPage(from) -> 시작 leaf
    B-Tree는 실제 데이터(값)가 전부 leaf 노드에만 있다. 내부(internal) 노드는 
    "어디로 내려갈 지" 알려주는 이정표일 뿐이다.
    그래서 스캔의 시작점을 찾으려면, 루트에서 from 키를 따라 내려가 from이 들어있을 (혹은 들어있어야 할) leaf 노드를 찾는다.
    이게 findLeafPage(from)이다.

    핵심은 범위의 시작점 leaf 하나만 트리 탐색으로 찾는다는 점이다. 나머지는 트리를 안탄다.

2. leaf 안에서 순회
    각 entry where key >= from:
        if key >= to: return result
        add (key, value)
    찾은 leaf 안의 엔트리들을 왼쪽부터 본다. leaf 안의 키들은 정렬되어 있다.
    - key >= from인 지점부터 수집을 시작 (시작 leaf라도 from보다 작은 키가 앞에 섞여 있을 수 있으니 걸러낸다)
    - key >= to를 만나면 -> 즉시 종료. 정렬되어 있으니 이 뒤는 전부 to 이상이라 볼 필요 없다. 이게 to가 exclusive(미포함)인 이유이다.
    - 그 사이면 (key, value)를 결과에 담는다.

3. leaf = leaf.auxPage(sibling)
    leaf 하나를 다 흝었는데 아직 to에 안닿았다면, 다음 데이터는 옆(오른쪽) leaf에 있다.
    여기서 B-Tree의 결정적 특징이 나온다. leaf 노드들끼리 linked List처럼 옆으로 연결되어 있다.
    auxPage(=sibling pointer, 형제 포인터)가 오른쪽 이웃 leaf를 가리킨다.
    그래서 다음 leaf로 갈 때 루트부터 다시 내려가지 않고, 포인터 하나만 따라가면 된다.
    ```
   [leaf A] → [leaf B] → [leaf C] → INVALID
   ↑ auxPage로 연결된 leaf 체인
   ```
   INVALID는 "오른쪽에 더 이상 leaf가 없다"(트리의 맨 끝)는 뜻이라 while 루프가 끝난다.
    전체 그림
   ```java
    
        [internal 노드들]
        /      |      \
   findLeafPage(from)로 여기 딱 하나만 트리 탐색
        ↓
    [leaf A] → [leaf B] → [leaf C] → INVALID
        └──────── sibling 포인터로 쭉 스캔 ────────┘
            key >= to 만나면 중간에 종료
    ```
    이 설계가 좋은 이유(성능)
    트리 탐색(루트 -> leaf)은 0(log N)이다. 만약 매 엔트리마다 트리를 다시 탔다면, 범위가 M개가 있을 때 0(M log N)
    하지만 이 알고리즘은 트리 탐색 1번 + slibling 포인터로 쭉 이동이라 0(log N + M)이다 펌위가 클수록 이 차이가 커지며, B-Tree가 관계형 DB의
    기본 인덱스 구조인 가장 큰 이유 중 하나가 바로 이 범위 스캔 효울이다.

# require
require는 Kotlin 표준 라이브러리의 인자 검증(precondition) 함수이다.
"이 조건이 아니면 잘못된 인자다"라고 막아서는 문지기이다.

동작
require(fromInclusive <= toExclusive) {"fromInclusive ($fromInclusive) > toExclusive($toExclusive)}
- 조건이 참이면 -> 아무 일 없이 그냥 통과
- 조건이 거짓이면 -> IllegalArgumentException을 던지고, 메시지로 뒤의 람다 {...} 결과를 사용
  즉 위 코드는 "fromInclusive가 toExclusive보다 크면 잘못된 호출이다 -> 예외"라는 뜻이다.
  범위 스캔에서 시작점이 끝점보다 크면 애초에 말이 안되니 막는다.

왜 메시지가 {} 람다인가?
require(condition) {"에러 메시지"}
메시지가 람다로 감싸져 있다. 이건 조건이 거짓일 때만 메시지 문자열을 만들기 위해서이다.
위 예시엔 $fromInclusive 같은 문자열 템플릿 (String 연결)이 들어있다. 만약 람다가 아니라 그냥 문자열이었다면 호출할 때마다 매번 이 문자열을
조립해야 한다.
- 검증에 통과하는 정상 케이스도 마찬가지이다. 람다로 넘기면 실제로 실패했을 때만 조립하니 낭비가 없다. (지연 평가, lazzy evaluations)

형제 함수들
require(x) -> IllegalArgumentException -> 인자(argument)가 잘못됨 - 외부에서 넘어온 값 검증
check(x) -> IllegalStateException -> 객체의 상태(state)가 잘못됨
assert(x) -> AssertionError -> 절대 일어나면 안되는 내부 가정(JVM 옵션으로 켜야 동작)
그리고 값이 null이 아님을 보장하는 변형도 있다:
- requireNotNull(x) -> x가 null이면 예외, 아니면 null 아닌 x를 반환
- checkNotNull(x) -> 같은데 IllegalStateException

이렇게 require형태로 메서드로 들어온 인자 검증을 하는 것을 **방어적 프로그래밍(fail-fast)**라고 부른다. 
잘못된 값이 들어오면 조용히 이상하게 동작하는 대신 **즉시 시끄럽게 터트려서** 버그를 호출 지점에서 바로 잡게 하는 전략이다.

# Pair 객체
Kotlin 표준 라이브러리가 주는 값 두 개를 한 덩어리로 묶는 아주 작은 객체이다.

만드는 방법
```java
val p = Pair(1, "hello") //생성자
val p2 = 3 to "world" // to 중위 함수 (관용적)
```
to는 앞에서 본 util처럼 중위 함수이다. 3 to "world"는 Pair(3, "world")와 완전히 같다.

값 꺼내기.
```java
val p = 1 to "hello"

p.first // 1
p.second // "hello"
```

두 값은 first, second로 접근한다.
타입도 각각 기업한다.
- 위 p는 Pair<Int, String> 타입이라 first는 Int, second는 String이다.

구조 분해 (destructuring) -> 이게 편하다.
```java
val (key, value) = someLeaf.entryAt(i) //entryAt이 Pair 반환한다면?
// key = pair.first, value = pair.second로 자동 분해
```

Pair는 구조 분해를 지원해서, 두 값을 각각 이름 붙은 변수로 한번에 받을 수 있다.
앞서 본 range scan에서 결과가 List<Pair<Key, Values>>라면 이렇게 순회한다.
```java
for ((key, value) in result) {
    println("$key = value")    
}
```
이처럼 함수가 값 두개를 반환하는 구조가 필요할 때 쓰인다.
kotlin은 자바처럼 "여러 값 반환"이 안되는데, Pair로 흉내 내는 것
```java
fun splitNode(): Pair<Node, Key> {
    // ...노드를 나누고
    return newNode to midNode()
}

val (node, key) = splitNode()
```
B-Tree split에서 "새로 생긴 노드"와 "부모로 올라갈 키"를 같이 돌려줄 때 사용하는 패턴

세 개면 Triple
값이 세 개면 형제 격인 Triple<A, B, C>가 존재한다 (first, second, third)/
Pair의 약점은 first/second라는 이름이 의미를 안알려준다는 것.
fun getUser(): Pair<String, Int> //first가 이름? 나이? 알 수 없다.

결국 값이 의미를 가지거나, 세 개 이상이거나, 여러 곳에서 재사용되면 -> Pair 대신 이름 있는 data class를 만드는게 훨씬 좋다.
data class User(val name: String, val age: Int)

그래서 대부분은 "함수 안에서 잠깐 쓰고 버리는 임시 묶음이면 Pair, 도메인에서 의미를 가지면 data class"
