# Spring framework Contrbute일지

/core/ResolvableType.java에서 
최상위 레벨 객체 TopCreate.create(I) -> TopController 대상으로 String이 되어버리는 문제가 있다.
실제 기대하는 값은 Long이다.

이때 메서드 내부에 중첩된 객체의 경우 Create.create(I) -> Controller 대상 -> Long으로 제대로 반환된다.

원본 코드를 보자.
```java
			Type ownerType = parameterizedType.getOwnerType();
			if (ownerType != null) {
				return forType(ownerType, this.variableResolver).resolveVariable(variableToCompare);
			}
			// Fallback: comparison by variable name, independent of generic declaration context.
			for (int i = 0; i < variables.length; i++) {
				if (ObjectUtils.nullSafeEquals(variables[i].getName(), variableToCompare.getName())) {
					return forType(typeArguments[i], this.variableResolver);
				}
			}
```
위의 코드인데 이때 ownerType != null인 경우 보통 도달하지 못해서 찾기 힘든 버그이다.

우선 버그 메커니즘을 보자.
1. resolveVariable에서 톱 레벨 파라미터화 타입은 getOwnerType()==null이기 때문에 identity 루프가 실패하면 owner 재귀를 건너뛰고 이를 폴백한다.
2. 이렇게 폴백으로 떨어지게 되면 무관한 형제 인터페이스의 통명 변수와 이름으로만 매칭을 한다.
```java
예시를 보자.
TopSearch.I: getName()="I", getGenericDeclaration()=interface TopSearch
TopCreate.I: getName()="I", getGenericDeclaration()=interface TopCreate
```
이렇게 위와 같이 두개의 리플랙션을 보면 위 2개의 형제 객체일 경우 .equals()로 비교하는데 
기본적으로 TopSearch.I와 TopCreate.I는 서로 매칭이 안되는게 정상이지만,
폴백으로 넘어가게 되면 .getName을 통해 문자열만 비교하여 "I"=="I"로 매칭되어서 TopCreate.I를 찾을 때 TopSearch의 I에 해당하는
클래스가 들어오게 되는 것이다.
이때 문제는 중첩 타입은 ownerType != null이라 owner 재귀에서 terminal return이 되지만 톱 레벨인 객체는 다르다.
그렇기 떄문에 이러한 폴백의 범위를 줄여서 이러한 문제를 발생하지 않도록 하는것이 이번 내용의 주요 내용이다.

이때 해당 폴백을 그냥 삭제하게 되면 기존의 narrow()테스트가 깨진다 그렇기 때문에 제거보단 해당 폴백의 대상을 좁히는 방향을 선택

## java.lang.reflect는 타입정보(메타데이터) 만 다룬다.
"raw type"이란
- ParameterizedType = TopSearch<String, Long> 같은 제네릭 타입 표기 자체를 나타내는 리플렉션 객체
- getRawType() = 거기서 <String,Long>을 벗진 바탕 클래스/인터페이스, 즉 TopSearch.class.
- getOwnerType() = 그 타입을 감싸는(중첩시킨) 바깥 타입. 톱레벨이면 감싼 게 없어서 null, 중첩이면 바깥 클래스
즉 "raw인 TopSearch"는 <>를 땐 TopSearch.class 그 자체이다.

```java
┌───────────────────────────────┬───────────────────────────────┬─────────────────────────────────┐
│                               │ getRawType() (raw = <> 뗀 것)  │ getOwnerType() (감싼 바깥 타입)    │
├───────────────────────────────┼───────────────────────────────┼─────────────────────────────────┤
│ 중첩 Search<String,Long>       │ interface OwnerProbe$Search   │ class OwnerProbe (바깥 클래스)     │
├───────────────────────────────┼───────────────────────────────┼─────────────────────────────────┤
│ 톱레벨 TopSearch<String,Long>   │ interface TopSearch           │ null                            │
└───────────────────────────────┴───────────────────────────────┴─────────────────────────────────┘
```

그 결과를 리턴하기에 아래 이름 폴백이 도달하지 못하고, 교차 매칭 사고도 안난다.
톱레벨의 경우 ownerType = null -> if를 스킵하고 그 아래 이름 폴백으로 떨어지고 TopSearch.I를 이름만으로 매칭되어 사고가 난다.

해당 Owner 재귀는  Outer<T>.Inner<U> 같은 중첩 제네릭에서 바깥 타입의 변수를 풀려고 있는 건데, 우연히 해당 폴백을 막아줘서 중첩은 멀쩡했던 것.

이를 재현해보는 테스트를 작성해보자.

```java
import javax.lang.model.type.TypeVariable;

void resolveTypeAgainstSameNamedVariablesInTopLevelDeclarations() {
    Type resolvedType = resolveType(
            method(TopCreate.class, "create", Object.class).getGenericParameterTypes()[0], TopController.class);
    assertThat(resolvedType).isEqualTo(Long.class);
}

void resolveMethodLevelTypeVariableIsNotShadowedByClassVariable() {
    Type resolvedType = resolveType(
            method(TopRepo.class, "convert", Object.class).getGenericReturnType(), TopStringRepo.class));
    assertThat(resolvedType).isInstanceOf(TypeVariable.class);
}
```

"TopCreate에서 이름이 create이고 파라미터가 (object)인 메서드를 리플렉션으로 찾아서 Method 객체를 돌려준다"

이떄 왜 3번째 인자가 Object.class create의 실제 선언은 O create(I body) -> 파라미터가 타입변수 I이다.
그런데 JAVA는 컴파일 시 제네릭을 지우는 (Type erasure) 탓에, 바이트코드 레벨 시그니처는 create(Object)가 된다.
그래서 이 메서드를 찾으려면 erase된 파라미터 타입 Object.class로 매칭한다.
(fixture에 <I,O>에 상한이 없으니 I는 Object로 erase된다.)

Object.class는 "메서드를 찾기 위한 검색 키"일 뿐, 우리가 해석하려는 타입이 아니다.

resolveType(대상, 컨텍스트 클래스) -> 두 인자의 의미
resolveType은 GenericTypeResolver.resolveType(Type genericType, Class<?> contextClass)이다.
- 1번 인자 = 해석하려는 타입(여기서 타입변수 I)
- 2번 인자 = "어느 클래스 관점에서 보나"하는 컨텍스트

resolveType의 시그니처가 (해석대상, 관점클래스) 순서일 뿐이다. I하나만 던지면 그게 뭔지 알 수 없고, TopController 관점에서
봐야 TopController implements TopCreate<Long, Long> -> I = Long이라는 바인딩이 생긴다.
