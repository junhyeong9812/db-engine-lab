BTreePage의 readIntAt 메서드와 writeIntAt메서드를 보자.

읽기 (readIntAt)
ByteBuffer.wrap(page.read(offset, 4)).int
1. page.read(offset, 4) -> 페이지 안에서 offset부터 4바이트를 잘라 ByteArray(4)로 반환(이 자체는 순수 Byte배열)
2. ByteBuffer.wrap(ByteArray(4)) -> 그 4Byte 배열을 감싸는 ByteBuffer로 "해석모드"에 들어간다 -> 배열을 복사하지 않고 그대로 참조한다.
3. .int -> 감싼 버퍼의 4Byte를 big-endian(ByteBuffer 기본 순서) Int로 해석해서 꺼낸다. -> 즉, Byte 배열 -> 의미 있는 Int로 디코딩한다.

쓰기 (writeIntAt)
page.write(offset, ByteBuffer.allocate(4).putInt(v).array())
1. ByteBuffer.allocate(4) -> 정확히 4Byte 크기의 새 빈 버퍼를 만든다.(capacity = 4, 아직 내용은 0)
2. .putInt(v) -> 그 4Byte 안에 v를 big-endian Int 형식으로 인코딩해 채워 넣는다.
3. .array() -> 그 결과물을 순수 ByteArray(4)로 꺼낸다.
4. page.write(offset, 그 배열) -> 페이지의 offset 위치에 그 4Byte를 그대로 덮어쓴다.

즉 "Int -> Byte 배열로 인코딩 -> 페이지의 정해진 위치(offset)에 write"가 맞다.
readLongAt/writeLongAt은 크기만 4->8Byte로 바뀐 완전히 동일한 패턴

핵심: Page는 순수 바이트 저장소(ByteArray 조각을 read/write)만 알고, Int/Long으로서의 "의미"는 결국
ByteBuffer가 인코딩/디코딩을 담당한다는 역할 분리이다.
-wrap은 기존 바이트를 해석할 때, allocate는 새 Byte를 만들어낼 때 사용한다.

해석 모드란?
ByteBuffer는 내부적으로 세가지 상태를 가진 커서 기반 뷰이다.
-> 커서 기반 뷰? -> 데이터를 담는 배열 위에 몇개의 위치 표시자(포인터)를 두고, 그 위치들을 이동시키면서 읽고 쓰는 방식이다.

- backing array -> 실제 Byte가 담긴 배열
- position -> 다음 읽기/쓰기가 일어날 위치(커서(포인터))
- limit -> 여기까지만 읽고/쓰기 가능한 경계
- capacity -> 배열 전체 크기

ByteBuffer.wrap(byteArray)가 하는 일:
- 그 ByteArray를 복사하지 않고 그대로 backing array로 삼는다.
- position = 0, limit = capacity = byteArray.size로 초기화
- 즉 "이 바이트 배열을 처음부터 끝까지 읽을 수 있는 상태"로 세팅만 해주는 것 -> 데이터 변환은 아직

.int (=getInt())가 하는 일:
- 현재 position부터 4Byte를 꺼내서, big-endian 순서로 해석해 Int로 조립
- 다 읽고 나면 position이 4만큼 이동한다.

-> 즉 해석모드란 wrap은 순서/경계만 세팅하는 준비단계이며, 실제 Byte 시점에 일어난다는 뜻이며,
코드에서는 매번 read(offset, 4)로 딱 4Byte만 잘라 새 Buffer를 wrap하고 .int 하나만 꺼내니,
Position 전진이나 재사용 걱정 없이 "이 4Byte = Int"라는 1회성 변환기로 사용한다.

반대로 allocate(4)는 backing array가 없는 상태로 시작(new byte[4] 내부 생성 + position = 0)해서,
putInt(v)가 그 위치에 4Byte를 써넣고 position을 전진시키는 것 -> 그래서 wrap은 "기존 바이트를 읽어내는 뷰",
allocate는 "새 바이트를 채워나가는 그릇이다"

여기서 "뷰(view)"란?
원본 데이터를 복사하지 않고, 그 위에 구조화된 방식으로 읽고 가지 성질이 핵심이다.
1. 소유권 없음. 참조만 -> ByteBuffer.wrap(byteArray)는 그 byteArray의 복사본을 만들지 않고, 같은 매 Buffer를 통해 값을 바구면
   원본 배열도 같이 바뀐다.(반대도 마찬가지) -> "새로운 데이터"가 아니라 "기존 데이터를 보는 창"이다.
2. 해석 방식만 얹음 -> 원본은 그냥 byte[4]일 뿐인데, 그 위에 ByteBuffer라는 view를 씌우면 "이건 4Byte"러눈 햐석 규칙(big-endian 조합 방식)이 추가된다.
    -> 데이터 자체는 그대로, 그걸 바라보는 방식만 달라지는 것

즉 우리가 구현중인 BTreePage도 이와 같은 패턴으로 Page가 들고 있는 원시 ByteArray 4096 바이트를 복사하지 않고,
그 위에 0번째 바이트는 type, 1~4번째는 keyCount...같은 구조적 의미를 씌워 읽고 쓰는 것, ByteBuffer가 바이트를 Int/Long으로
해석하는 뷰이 듯 BTreePage는 페이지 전체를 BTree 노드로 해석하는 뷰이다.
