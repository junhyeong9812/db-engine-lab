## forEachIndexed 메서드
inline fun <T> Iterable<T>.forEachIndexed(action: (index: Int, T) -> Unit)
-> 컬렉션을 앞에서부터 돌면서 (몇 번째인지, 그 원소) 두 개를 람다에 넘겨주고, 반환값은 없다.(Unit) 
그냥 forEach는 원소만 주는데, 인덱스도 같이 필요할 때 쓰는 메서드이다.

1) forEachIndexed
```java
keys.forEachIndexed{i, k -> idx.insert(k, (i * 1000).toLong())}
```

2) 전통적인 인덱스 루프
```java
for (i in keys,indices) {
    val k = keys[i]
    idx.insert(k, (i * 1000).toLong())
}
```

3) withIndex + 구조분해 (테스트의 검증 루프가 쓰는 방식)
```java
for((i, k) in keys.withIndex()) {
    idx.insert(k, (i * 1000).toLong())    
}
```
3가지가 같은 기능을 하는 구조이며, forEachIndexed는 inline 함수라 람다가 호출 지점에 펼쳐져서 객체 생성 오버헤드가 없고, 대신 람다 안에너는 break/continue를 쓸 수 없다는 점이다.(return@forEachIndexed로 그 회차만 건너뛰는 건 가능).
루프 도중 빠져나가야하면 일반 for를 쓰는게 맞다.
이런 람다 이름 규칙에는 mapIndexed(변환해서 새 리스트), filterIndexed 처럼 indexed 접미사가 붙으면 "인덱스를 같이 넘겨주는 버전이다."

###
```java
@Test
    fun `정렬 안된 순서로 insert 해도 정렬된 상태 유지`(@TempDir tempDir: Path) {
        val path = tempDir.resolve("btree.db").toString()
        PagedFile(path).use { pf ->
            BufferPool(pf, capacity = 16).use { bp ->
                val idx = BTreeIndex(pf, bp)
                val keys = listOf(50L, 10L, 80L, 30L, 70L, 20L, 60L, 90L, 40L)
                keys.forEachIndexed { index, key -> idx.insert(key, (index * 1000).toLong())  }

                for ((index, key) in keys.withIndex()) {
                    assertEquals((index * 1000).toLong(), idx.search(key), "key=$key")
                }
            }
        }
    }
```
해당 테스트의 의의
삽입 순서가 정렬과 무관해도 각 키가 자기 값을 그대로 들고 있는가?를 보는 테스트이다.
이게 왜 의미가 있는 지 보면, B-Tree는 삽입할 때마다 페이지 안에서 ㅣ를 정렬 위치에 끼워 넣으려고, 기존 엔트리들을 뒤로 밀어낸다.(shift), 이때 키 배열만 밀고 값 배열을 안밀거나, 한 칸 어긋나게 밀면 "키 50 자리에 원래 30의 값이 들어가 있는"식의 오프바이윈 버그가 생긴다.
순ㅅ대로 넣는 테스트는 shift가 아예 안일어나서 이 버그를 못 잡지만, 50 -> 10 -> 80처럼 뒤섞어 넣으면 shift가 강제로 발생한다. 그래서 값을 i * 1000으로 키마다 다르게 준거다.
전부 값은 값이였다면 엉뚱하게 매핑되어도 통과된다.

### 오프바이윈(off-by-one) 버그
인덱스나 개수가 딱 1만큼 어긋나서 생기는 오류이다.
경계 조건에서만 터지기 때문에 대충 돌려보면 멀쩡해 보이는게 특징이다.
대표적인 형태가 set이다.
1) < 와 <= 혼동 -> 배열 크기가 10인데 for(i in 0..10)처럼 돌면 마지막에 인덱스 10을 건드려서 범위 밖 접근이 난다. 반대로 0 until 0로 쓰면 마지막 원소를 빼먹는다.
2) shift(밀어내기) 방향/범위 실수 -> B-Tree 삽입에서 실제로 나오는 형태이다. key 배열이 [10,30,50]인데 20을 인덱스 1자리에 끼워 넣으려면, 1번 이후 원소들을 뒤에서부터 한 칸씩 밀어야 한다.
```java
// 올바른 방향: 뒤에서 앞으로
for (i in count downTo pos + 1) {
    keys[i] = keys[i - 1]
    values[i] = values[i - 1] //값 배열도 같이 밀어야 한다.
}
keys[pos] = newKey
values[pos] = newValue
```

여기서 시작을 count - 1로 잡으면 마지막 원소가 덮어씌워져 사리지고, 앞에서 뒤로 돌면 같은 값이 뒤로 계속 복사되며 뭉개진다. 그리고 values 줄을 빼먹거나 values[i] = values[i]처럼 오타를 내면 -> 앞서 말한 "키 50 자리에 30의 값이 붙어 있는" 상태가 된다.
키만 정렬돼 보이니까 육안으로는 정상처럼 보이고, 그래서 search(50) == 0 같은 매핑 단언이 필요하다.

3) 이진 탐색의 경계 -> lo, hi, mid 갱신에서 mid vs mid + 1을 잘못 쓰면 무한 루프에 빠지거나, 찾아야 할 삽입 위치를 한 칸 앞/뒤로 알려준다. B-Tree의 findSlot 계열 함수가 늘 이 위험을 안고 있다.
이름의 유래는 "울타리 기둥 문제"로, 10m 울타리를 1m 간격으로 세우면 구간은 10개지만 기둥은 11개가 필요하다, 개수와 경계를 헷갈리는 순간 딱 1이 틀어진다.
4) 방버법은 결국 경계값을 직접 때리는 테스트이다.
