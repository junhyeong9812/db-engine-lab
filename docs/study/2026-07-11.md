companion object란?
Kotlin에는 Java의 static 키워드가 없다. 대신 클래스 안에 딱 하나 둘 수 있는 특별한 object가 companion object(동반 객체)이다.
"이 클래스와 짝을 이루는, 인스턴스가 필요 없는 맴버들의 집"이다.
instance member와 companion member를 비고해보면,
```java
class BufferPool(): {
    fun flushAll() {}
    companion object {
        const val DEFAULT_CAPACITY: Int = 256
    }
}
```
위와 같이 2개이 맴버가 있다고 보면 
flushAll() -> 인스턴스 맴버   -> 접근법:bufferPool.flushAll() -> 특정 인스턴스의 상태(pages)를 다루므로 인스턴스가 있어야 함
DEFAULT_CAPACITY -> companion 맴버 -> 접근법: BufferPool.DEFAULT_CAPACITY -> 인스턴스와 무관한 공용 값 -> 클래스 이름으로 접근

이에 대해 한가지의 예를 보면 결국
className.something()을 쓰는 순간 kotlin은 오직 companion object 안에서만 그 맴버를 찾는다.
이처럼 companion이란 해당 객체의 전역 상수나 전역 변수로 도메인 지식을 담은 객체를 넣어 공유할 수 있으며, 
이러한 companion은
- 상수
- 팩토리 메서드
- 인스턴스가 필요 없는 유틸 함수
에서 자주 사용된다.


closeable이란?
java.io.Closeable은 자바 표준 인터페이스로, 딱 하나의 메서드를 요구한다.
fun close()
"나는 다 쓰고 나면 정리(닫기)가 필요한 자원"이라는 계약이다.
파일 핸들, 소켓, DB 커넥션 처럼 GC만으로만 제때 정리되지 않는 OS자원을 가진 객체를 구현
이때 closeable을 구현하면 kotlin의 use 함수를 사용할 수 있으며,
```java
BufferPool(pagedFile).use { pool ->
    val page = pool.newPage()
    // ... 작업 ...
}   // 블록을 벗어나면 예외가 나든 정상 종료든 pool.close()가 자동 호출됨 → flushAll() 보장
```
자바의 try-with-resources와 같은 것으로, close() 호출을 잊거나 예외 때문에 건너뛰는 실수를 막아주며, 리소스 누스(파일 안 닫힘, 데이터 flush안됨) 방지의 표준 패턴이다.

한 장 요약

companion object  →  Java의 static 대체. 클래스이름.멤버 로 접근.
인스턴스 상태와 무관한 상수·팩토리를 여기 둔다.
→ 그래서 인스턴스 메서드 flushAll을 BufferPool.flushAll()로 부르면 에러.

Closeable         →  "정리가 필요한 자원" 계약. close() 하나 구현.
여기선 close() = flushAll()(dirty page 디스크 반영).
use { } 로 자동 close 보장 → 리소스
