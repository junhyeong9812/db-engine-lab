val -> final 참조, 한번 할당되면 그 참조는 다시 못바꾼다.(단, 참조가 가리키는 객체 내부 상태는 바꿀 수 있다.-> val list = mutableListOf(1)이면 list = other는 안되지만 list.add(2)는 된다.
const val - 컴파일 타임에 값이 확정되기 때문에, 바이트 코드 컴파일 시 그 값 자체가 사용되는 곳마다 리터럴로 그대로 박혀서 들어간다(즉 인라이닝), -> java의 static final 사수와 동일한 방식이다.


BTreePage.companion object의 상수의 의미
각 상수는 디스크 페이지 4096바이트를 어떻게 나눠쓸 지 정의하기 위한 상수이다.

페이지 레이아웃 (총 Page.PAGE_SIZE = 4096 bytes)이며
type (1byte) -> LEAF(0)/INTERNAL(1)
keyCount (4byte) -> 현재 저장된 엔트리 개수
auxPage (4byte) -> LEAF면 다음 leaf 페이지 번호, INTERNAL이면 맨 왼쪽 child
parentPage (4byte) -> 부모 페이지 번호
------------------------------------> HEADER_SIZE -> 13
entry 0: key(8) + value(8) = 16
entry 1: key(8) + value(8) = 16
....
entry 254 (마지막)

- INVALID = -1 -> "페이지 없음"을 뜻하는 sentinel 값, root의 parentPage나 마지막 leaf의 auxPage(다음 leaf 없음) 같은 곳에 씀, initAsEmpty의 기본 인자로 쓰이는게 그 예시.
- KEY_SIZE = 8 -> key를 Long으로 저장하니 8바이트
- VALUE_SIZE = 8 -> value도 8바이트. leaf에선 실제 레코드 값(또는 포인터), internal 노드였다면 child page번호가 여기 들어갈 자리(현재는 leaf-only 단계라 값으로만 쓰인다.)
  -ENTRY_SIZE = KEY_SIZE + VALUE_SIZE = 16 -> 엔트리 하나(key + value 쌍)가 차지하는 바이트 수.
  keyAt/valueAt/setEntry가 HEADER_SIZE + slot * ENTRY_SIZE로 위치를 계산하는 근거가 된다.
- HEADER_SIZE = 1 + 4 + 4 + 4 = 13 ->위 레이아웃 그대로: type(1) + keyCount(4) + auxPage(4) + parentPage(4).이 뒤부터 엔트리 영역이다.
- MAX_ENTRIES = (4096 - 13)/16 = 255 - 헤더를 뺀 나머지 공간에 16바이트 엔트리가 몇 개 들어가는 지.
  insertAt의 require(count < MAX_ENTRIES), isFull()이 이 값으로 페이지 꽉 참을 판정
