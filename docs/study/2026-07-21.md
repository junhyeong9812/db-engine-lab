# Kotlin의 sealed란?
sealed란 Kotlin에서 "이 클래스의 하위 타입은 여기 정의된 게 전부다"라고 컴파일러에 못박는 키워드

일반 open class는 누구든 어디서든 상속할 수 있지만, sealed class는 하위 타입을 같은 파일(정확히 같은 패키지 모듈) 안에서만 정의할 수 있다.
그래서 컴파일러가 "하위 타입 목록이 닫혀 있다(sealed = 봉인됨)"는 걸 알게 된다.

```java
sealed class Constraint {
    data class PrimaryKey(val columns: List<String>) : Constraint()
    data class Unique(val columns: List<String>) : Constraint()
    data class ForeignKey(/* ... */) : Constraint()
}
```
여기서 일반 class인 경우 Constraint()에서 오류가 나오는데
이유는 Kotlin 클래스가 기본적으로 final(상속 금지)이기 때문이다.
java에서는 클래스가 기본으로 상속 가능하다고 막으려면 final을 붙이지만, kotlin은 정반대이다.
그냥 class Constraint라고 쓰면 자동으로 final이라서, 다른 클래스가 Constraint()로 상속하려는 순간 오류가 나는 것.
"This type is final, so it cannot be inherited from"
상속을 허용하려면 셋 중 하나를 선언 해야된다.
keyword -> mean
open -> 상속 가능 + 자기 자신도 인스턴스화 가능
abstract -> 상속 가능 + 자기 자신은 인스턴스화 불가
sealed -> 상속 가능하지만 같은 패키지,모듈 안에서만 + 자기 자신은 인스턴스화 불가 (암묵적으로 abstract)
그렇기 때문에 sealed는 하위 타입 목록을 봉인하는 기능과 상속 허용까지 되어있는 키워드라 보면 된다.


이게 왜 유용하냐면, when 분기에서 컴파일러가 모든 경우를 다뤘는 지 검사해줄 수 있기 때문이다.
```java
when (c) {
    is Constraint.PrimaryKey -> { /* tag 0 */ }
    is Constraint.Unique     -> { /* tag 1 */ }
    is Constraint.ForeignKey -> { /* tag 2 */ }
    // else 불필요 — 전부 다뤘다는 걸 컴파일러가 앎
}
```
만약 나중에 Check 제약을 추가하면 기존의 모든 when 분기가 컴파일 에러로 바뀐다.
"새 제약을 추가했는데 직렬화 코드에서 처리 안 함" 깉은 버그가 런타임이 아니라 컴파일타임에 잡히는 것이다.
이때 enum과 비교하면: enum은 "값 목록이 고정", sealed는 "타입 목록이 고정"이다. 제약마다 들고 있는 데이터 구조가 다르기 때문에
(PK는 컬럼 목록만, FK는 참조 테이블 컬럼까지)각각을 별도 data class로 만들 수 있는 sealed를 사용하는 것

# filterIsInstance<T>()는 kotlin 표준 라이브러리 함수로, 리스트에서 특정 타입인 요소만 골라내는 필터이다.
일반 filter로 쓰면 이렇게 된는걸:
```java
constraints.filter {it is Constraint.PrimaryKey}
```
위 코드를 한번에 처리해준다.
```java
constraints.filterIsInstance<Constraint.PrimaryKey>() 
```
List<Constraint.PrimaryKey>
핵심 차이는 반환 타입이다. filter는 걸러내도 결과가 여전히 List<Constraint>라서, 요소의 PK 전용 속성에 접근하려면 캐스팅이 필요하다.
filterIsINstance는 걸러냄과 동시에 List<Constraint.PrimaryKey>로 타입까지 좁혀줘서 바로 .columns 같은 속성을 쓸 수 있다.

validateConstraints()에서 이걸 쓰는 이유는 명확하다:
val pks = constraints.filterIsInstance<Constraint.PrimaryKey>()

constraints는 PK/Unique/FK가 섞인 List<Constraint>인데, 여기서 PrimaryKey만 뽑아서 "PK가 2개 이상 선언되지 않았나"를 검사하려는 것이다.
pks.size > 1이면 잘못된 스키마인거다.
동작 원리 자체는 단순하다. -> 리스트를 순회하면서 각 요소에 is 검사를 하고, 통과한 것만 새 리스트에 담아 반화한다.
앞서 본 sealed class의 하위 타입 구조와 조합하면 "섞인 리스트에서 특정 종류만 꺼내 검사"하는 패턴이 캐스팅 없이 깔끔하게 된다.

# withIndex() 메서드
withIndex()는 List를 (Index,Value) 쌍의 시퀸스로 바꿔주는 표준 라이브러리 함수

정확히는 각 요소를 IndexedValue<T>(index와 value 두 속성을 가진 data class)로 감싼 Iterable을 반환한다. 보통 for 루프에서 구조 분해(destructuring)와 같이 쓴다.
```java
for ((i, col) in schema.columns.withIndex()) {
        // i = 인덱스, col = ColumnDef
}
```
java에서 for (int i = 0; i < list.size(); i++)로 인덱스를 따로 관리하던 걸 대체하는 관용구이다.
인덱스가 필요 없으면 그냥 for (col in schema.columns)를 쓰고, 인덱스까지 필요할 때만 withIndex()를 쓰면 된다.
비슷한 메서드로는 forEachIndexed { i, col -> ...}도 있는데, 같은 일을 람다 스타일로 하는 것뿐.
Python의 enumerate()와 같은 역할이다.

# Tuple의 decode와 encode
우선 Tuple의 decode와 encode의 계약은 바이트 레이아웃이며, 이를 알아야 해당 코드를 쉽게 알 수 있다.
바이트 레이아웃이란?
[NULL bitmap (ceil(n/8) byte) ][non-null 값들, 컬럼 순서대로]
- Null인 컬럼은 값 영역에 아예 안 쓰인다(0byte). 대신 bitmap의 해당 비트가 1로 켜진다.
- 값 영역에는 타입 정보가 없다. "몇 번째 값이 무슨 타입인지"는 전적으로 schema가 알려주기 때문에, decode에 schema를 반드시 넘겨야 한다.

## 공통 비트 연산
(n + 7) / 8은 정수 나눗셈으로 올림(ceil)을 구하는 광용구이다. 컬럼이 9개면 (9+7)/8 = 2byte -> bit 9개를 담으려면 2byte가 필요하다.
컬럼 i의 비트 위치는 두 좌표로 쪼개진다.
- i/8 -> 몇 번쨰 byte인지
- i%8 -> 그 byte 안에서 몇 번째 byte인지

예: 컬럼 10 -> byte 1의 bit 2

즉 데이터는 
- bitmap은 데이터 유무(정확히는 Null 여부 -> 비트 1 = null)만 표현한다.
- 몇 번째 비트가 어느 컬럼인지는 schema에 정의된 컬럼 순서로 정해진다.
- 실 데이터는 바이트 배열의 값 영역에 있으며, schema는 그 byte를 어떻게 읽을 지만 알려준다.

역할을 보면
bitmap(bytes 앞부분): 컬럼 null 여부
값영역(bytes뒷부분): 실제 데이터(그냥 이어붙인 byte 덩어리)
schema: 해석 규칙, 컬럼 순서, 각 컬럼의 타입(= 몇 바이트를 어떤 방식으로 읽을 지)

decode가 돌아가는 방식으로 다시 말하면: 값 영역은 구분자 없는 바이트 덩어리라서 혼자는 못자른다.
schema가 0번째 컬럼은 "Int니깐 4바이트, 1번째는 String이니깐, 길이 프리픽스 읽고 그만큼"하고 자를 위치를 알려주고, bitmap이 "이 컬럼은 null이니깐, 값 영역에서
건너뛰지도 말고 아예 읽지마"하고 스킵 여부를 알려주는 것이다.

한 문장으로 정리하면: bitmap = null 여부, 값 영역 = 실 데이터, schema = 그 바이트를 자르고 해석하는 설계도라 볼 수 있다.
