# Kotlin의 init
- init 블록은 객체가 생성될 때(생성자가 실행될 때) 딱 한번 자동으로 실행되는 코드 조각이다.
  (이름이 붙은 함수처럼 나중에 obj.init()으로 호출할 수 있는 게 아니다. 호출하는 주체는 항상 "생성자"이며, 우리가 직접 부를 수 없다.)

kotlin에서는 왜 생성자에 초기화 동작을 넣는게 아닌 init에 넣어야 될까?
- 주 생성자(primary constructor)는 class xxxx(...)의 괄호 부분인데, 여기는 코드를 넣을 자리가 없으며, 파라미터 선언만 가능하다.(그래서 "생성 시점에 실행할 로직"을 넣을 공간이 바로 init 블록이다.)
즉, kotlin에서는
- 주 생성자 괄호 -> 파라미터 받기(...)
- init 블록 -> 그 파라미터를 가지고 실행할 초기화 로직{...}
java로 치면 생성자 본문(public xxxx(...) {...})이다.

## 실행 순서
kotlin에서는 클래스 본문은 위에서 아래로 순서대로 실행되며, 프로퍼티 초기화와 init 블록이 선언된 순서대로 섞여서 실행된다.
```java
class Example(name: String) {
    val a = "1".also { println("프로퍼티 a") }   // 1
    init { println("init 블록 1") }              // 2
    val b = "2".also { println("프로퍼티 b") }   // 3
    init { println("init 블록 2") }              // 4
}
// 출력: 프로퍼티 a → init 블록 1 → 프로퍼티 b → init 블록 2
```
주의점: init에서는 자기보다 아래 선언된 프로퍼티를 못쓴다.
init은
1. 생성 시점의 검증(validation)
2. 파생 상태 준비
3. 여기 코드처럼 "생성과 동시에 반드시 해둬야 하는 부수 작업"
등에서 많이 쓰인다.


여러 개 vs 팩토리

- init 블록은 여러 개 둘 수 있습니다(위 예제처럼). 순서대로 실행된다. 보통은 하나로 몰지만, 프로퍼티 선언 사이사이에 로직이 필요하면 눈다.
- 초기화가 실패할 수 있거나(예: 다른 결과 객체를 반환하고 싶거나) 복잡하면, init 대신 companion object의 팩토리 메서드를 쓰는 게 나을 때도 있다. init은 "무조건 이 객체를 만든다" 경로에서만 돈다.
---
요약

init 블록  =  "메서드"가 아니라, 생성자가 실행될 때 자동으로 도는 초기화 코드.
- 직접 호출 불가. new 할 때 한 번만.
- 주 생성자 괄호엔 로직을 못 넣으니, 그 로직의 자리가 init.
- 프로퍼티 초기화와 "선언 순서대로" 섞여 실행 → 아래 프로퍼티는 못 씀.
- 주 용도: 검증 / 파생 상태 준비 / 생성 시
