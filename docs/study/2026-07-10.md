BTreePage -> mid = (lo + hi) ushr 1
ushr를 사용하는 이유
1. 덧셈 자체는 비트를 안 잃는다.
   lo, hi는 배열 인덱스라 둘 다 0 ~ 2^31-1 범위의 음이 아닌 Int이다.
   둘의 진짜 합은 최대 2^32 - 2가 될 수 있다.
   2(2^31-1)
- 이건 signed Int(최대 2^31-1)로는 넘치지만
- 32비트 자체로 보면 안넘친다. (unsigned로 보면 2^32-1까지 표현 가능하기 때문에, 진짜 합은 2^32-2이하
  즉 lo-hi가 오버플로우해서 음수로 "보이더라도", 32비트 안에는 진짜 합의 모든 비트가 그대로 살아있다.
  단 최상위 비트를 부호로 해석해서 음수처럼 보일 뿐이다.

2.ushr은 그 비트를 unsigned로 해석해서 나눈다.
ushr(unsigned shift, 자바의 >>>)은 부호 비트를 무시하고 왼쪽으로 채운다.
그래서 그 32비트를 unsignedㄱ밧 그대로 2로 나눈 결과를 준다.

이를 예시로 표현해보면
lo = hi = 2_000_000_000(20억) 둘 다 Int max 2_147_483_647보다 작다.
진짜 값 (4_000_000_000) / 2 = 2_000_000_000
lo + hi (Int) -> -294_967_296 (= 4_000_000_000 - 2^32)
(lo + hi) / 2 -> -147_483_648
(lo + hi) shr 1 (부호 >>) -> -147_483_648 (부호 유지)
(lo + hi) ushr 1 (>>> ) -> 2_000_000_000
즉 부호비트 위치까지 표현에 사용하는 방식으로 오버플로우를 방어한다.

kotlin -> require method
require란? 함수 인자에 대한 사전 조건 검증
시그니처는
```java
inline fun require(value: Boolean, lazyMessage: () -> Any): Unit {
    if(!value) {
        throw IllegalArgumentException(lazyMessage().toString())
    }
}
```
* inline 예약어
-> require 같은 고차 함수(람다를 받는 함수)의 성능 함정을 없애는 키워드
-> 기본 개념: 호출 대신 "붙여넣기"
-> 보통 함수는 호출하면 -> 스택 프레임 만들고 -> 점프하고 -> 리턴하는 비용이 있는데. inline을 붙이면 컴파일러가 호출하는 자리에 본문을 그대로 복사해 넣는다. 즉 함수호출이 사라진다.
BtreePage의 44줄을 보면
```java
require(count < MAX_ENTRIES) { "btree page full (max=$MAX_ENTRIES" }
```
위와 같은 함수가 있다고 했을 때 
원래는 require 메서드의 포인터를 통해 스택프레임을 만들고 호출 위치에 대한 포인터 좌표를 등록하고 점프하고 리턴하고 이러한 걸
```java
if (!(count < MAX_ENTRIES)) {
    throw IllegalArgumentException("btree page full (max=$MAX_ENTRIES")
}
```
위와 같이 컴파일 시점에 이렇게 변환시켜서 함수 호출 비용을 없애고, 그 내용이 그 자리에 박히도록 할 수 있다.

그런데 이떄 require라는 함수 호출을 없애는 건 좋은데 
진짜 이유는 "람다"때문이다.
inline이 정말 중요해지는 건 **함수가 람다를 인자로 받을 때**이다.
만약 람다가 인라인이 아니라면, () -> {"btree page full ..."}는 컴파일 시 익명 객체(Function 인스턴스)로 힙에 만들어진다.
require를 부를 때마다 람다 객체가 하나씩 생성되어 GC 부담이 생겨난다.
inline이면 그 람다 본문도 호출 지점에 직접 펼쳐져서, 람다 객체가 아예 안만들어진다.
-> require가 정상 통화할 때는 조건 검사 한줄만 남기고, 문자열 보관도 객체 생성도 전혀 없다.
-> 이게 지연 평가 + 무비용을 가능하게 한다.

inline의 역할 세가지
1. 함수 호출 오버헤드 제거 -> 분문을 호출부에 복사
2. 람다 객체 생성 제거 -> 고차 함수의 진짜 이득(GC 압력 감소)
3. return, reified등 부가 기능 활성화 

인라인이라 가능해지는 것:
- non-local return: 인라인 람다 안에서는 바깥 함수를 통째로 return 할 수 있다. 예를 들어 list.forEach{if (it < 0) return }에서
  retrun이 forEach가 아니라 감싸는 함수를 바져나간다. 이건 forEach가 inline이라 가능한 것이다.
```java
fun a() {
    list.forEach { if (it < 0) return }
    println("done")
}
```
위와 같은 메서드가 있을 때 실제 return은 a()자체를 나가도록 설계할 수 있어서 forEach만 벗어나는게 아닌 순수 메서드 자체도 나갈 수 있다.
즉 음수를 만나는 순간 forEach도 멈추고, 그 그 아래 done로 실행안되고 a 함수를 종료시킬 수 있다.
non-local(비지역) return이라 부르는 이유가 바로 이것이다. return이 자기가 속한 람다(지역)가 아니라, 그 람다를 감싸는 바깥 함수(비지역)을 탈출할 수 있다. forEach가 inline이라서 람다 본문이 a()안에 그대로 펼쳐지고, 그러니 return은 진짜 a()의 리턴이 된다.
이때 실제 forEach만 빠져나가고 싶은 경우 return@forEach같은 형태로 사용해서 이번 반복만 스킵하여 continue처럼 사용한다.


- reified 타입 파라미터: inline fun <reified T> ...처럼 제네릭 타입 T를 런타임에 실제로 알 수 있게 해준다->즉 표현은 T지만 실제 class의 타입이 빌트인된다. (T::class,is T 가능). 
  JVM의 타입 소거를 우회한다. 본문이 호출부에 펼쳐지니깐 실제 타입이 그자리에 박혀서 가능한것.
```java
inline fun <reified T> printType() = println(T::class.simpleName)
-> 컴파일 후
printType<String>() // 컴파일 후 -> println(String::class.simpleName)로 펼쳐진다.
printType<INt>()   // 컴파일 후 -> println(Int::class.simpleName)으로 펼쳐진다. 
```
위와 같이 각 호출 지점마다 T가 String, Int로 각각 확정된 채 복사된다. (보통의 함수 본문은 딱 한번만 컴파일 되는데, 그 한번 안에 T가 뭔지 런타임에 알 수 없으니 T::class가 불가능한 것,)

* 타입 소거 -> 런타임엔 T 정보가 아예 지워져서 사라진다. 소거(erasure)이라는 이름 그대로 사라진다.
타입 정보는 컴파일 타입에만 존재하고, 런타임엔 버려진다.
* 제네릭 타입 T를 아는건 컴파일러뿐이다. 컴파일이 끝나고바이트 코드가 되면 T는 Object(또는 상한 타입)로 치환되어 없어진다. -> 런타임에는 이게 원래 무슨 탕비이였는지 정보가 없다.
예시로 보면
```java
fun <T> foo(x: T) {
    // 여기서 T::class 하려고 하면,
}

//컴파일 후 이 함수는 JVM 입장에서 사실 상 동작을 보면
void foo(Object x) {
    // T는 Object로 소거된다.
    // T가 어떤거였는지 알 수 없다.
}
```
-> 즉 foo<String>()을 부르든 for<Int>()를 부르든 같은 이 한벌의 바이트코드가 실행된다.
함수 안에서는 자기가 String으로 불렸는 지 Int로 불렸는지 런타임에서 알 수 가 없다.
T::class -> 이미 존재하는 클래스를 가리키는 참조(핸들)를 가져오는 것.
::class가 반환하는 것 -> T::class의 결과는 KClass<T> 타입의 객체 하나이다. 이건 "그 클래스에 대한 메타정보 헨들"이다. 클래스나 인스턴스가 아닌 리플렉션(reflection)토큰이라 생각하면 된다.
```java
val k: KClass<String> = String::class
println(k.simpleName)   //"String"
println(k.qualifiedName) // "Kotlin.String"
```
여기서 String::class는 String이라는 클래스 자체(이미 JVM에 로드돼 있는)를 설명하는 카드 한 장을 꺼내오는 것.
즉 reified와 연결하면,
inline fun <reified T> typeName(): String = T::class.simpleName ?: "?"
typeName<String>() // 컴파일 후 -> String::class.simpleName ->"String"
reified 덕에 T자리에 실제 타입(String)이 박히고, String::class가 이미 존재하는 String 클래스의 메타 핸들을 반환해서 이름을 꺼낸다.
여기서도 새로 생성되는 건 아무것도 없다.

그럼 다시
```java
inline fun require(value: Boolean, lazyMessage: () -> Any): Unit {
    if (!value) {
        throw IllegalArgumentException(lazyMessage().toString())
    }
}
```
로 돌아가서 핵심 특성을 보면
1. 메시지가 람다({...})인 이유 -> 지연 평가
2. inline + lazyMessage: () -> Any라서, 메시지 문자열은 조건을 실패했을 때만 만들어진다. 정상 통과할 땐 문자열 조합 비용("...$MAX_ENTRIES"같은 보관)이 아예 안들며, 그래서 require(cond, "msg")가 아닌 require(cond) {"msg"}형태를 사용
3. line 45의 require(slot in 0..count) -> 메시지 없는 오버로드
```java
public inline fun require(value: Boolean): Unit {
    require(value) {"Failed requirement."}
}
```
메시지를 생략하면 기본 문구 "Failed requirement"가 들어가서 여기서 slot이 0..count 범위 밖이면(음수거나 count 초과) 예외 및 메시지를 던진다.

형제 함수들과의 구분
kotlin엔 비슷한 검증 함수가 셋 있는데 던지는 예외 타입과 의미가 달라진다.
require: 
- 실패 시 예외: IllegalArgumentException
- 의미: 인자가 잘못됨 - 호출자 잘못(public 함수 입구)
check:
- 실패 시 예외: IllegalStateException
- 의미: 객체 상태가 잘못됨 - 지금 이 연산을 할 상태가 아님
assert:
- 실패 시 예외: AssertionError
- 의미: 내부 불변식 - 게다가 JVM -ea 옵션 없으면 아예 안 돈다.

insertAt에서 require를 쓴 건 "호출자가 넘긴 조건(슬롯 범위, 가득참 여부)이 계약 위반"이라는 뜻이야. 만약 "이 페이지는 지금 삽입 가능한 상태여야 한다"는 뉘양스면 check가 더 맞고, 여기선 인자 검증이니 require가 적절하다.
