# java.nio.ByteBuffer && java.nio.StandardCharsets
java.nio -> New I/O
데이터를 디스크에서 저장할 때 바이트 형태로 바꿀 때 쓰는 도구

## 1. ByteBuffer -> 바이트를 담는 상자
DB는 결국 모든 바이트(byte)로 디스크에 쓴다. Int, Long, 문자열.. 이 값들을 바이트 배열로 바꾸고(직렬화).
다시 읽어서 원래 값으로 복원(역직렬화)해야 한다.
ByteBuffer는 이 변환을 다루는 상자이다.

```java
val buffer = ByteBuffer.allocate(16) // 16바이트짜리 상자 생성
buffer.putLong(100L)    // Long(8바이트) 쓴다. -> 커서가 8로 이동
buffer.putLong(200L)    // 또 Long 씀 -> 커서가 16으로 이동

buffer.flip() // 쓰기 모드 -> 읽기 모드 전환

val a = buffer.getLong() // 100L 읽음
val b = buffer.getLong() // 200L 읽음
```
핵심 개념: 내부 커서 (position)

ByteBuffer의 가장 중요한 특징은 내부에 "지금 어디까지 읽었/썼는지"를 기억하는 커서가 있다는 것,
putLong을 하면 자동으로 커서가 8칸 뒤로 가고, 다음 putLong은 그 뒤에 이어서 쓴다.
앞에서 본 B+Tree leaf에 엔트리를 순서대로 밀어넣던 것과 비슷한 내용

flip() -> 커서를 맨 앞으로 이동
쓰기가 끝나고 읽으려면, 커서를 맨 앞으로 되돌려야하는데 이때 해당 동작을 해주는 메서드
이걸 안하면 커서가 맨 끝(16)에 있어서 읽을 게 없다.

타입별 메서드
- Int(4byte) : putInt/ getInt
- Long(8byte) : putLong/ getLong
- Short(2byte) : putShort/ getShort
- 단일 byte(1byte) : put / get
- 바이트 배열 통째로(가변) : put(byteArray)

Long이 왜 8바이트인지, getLong이 왜 정확히 그 위치의 8바이트를 읽어 복원할 수 있는지 — 이게 DB 페이지 레이아웃 설계의 기초.

## 2. StandardCharsets - 문자열 <-> 바이트 변환 규칙
ByteBuffer는 숫자는 잘 다루는데, 문자열은 "어떤 규칙으로 바이트로 바꿀 지"를 정해줘야한다. 그 규칙(문자 인코딩)을 상수로 안전하게 제공하는 게 
"StandardCharsets"이다.
```java
val text = "hello"

// 문자열 -> 바이트
val bytes: ByteArray = text.toByteArray(StandardCharsets.UTF_8)

// 바이트 -> 문자열
val restored = String(bytes, StandardCharsets.UTF_8)
```
왜 인코딩을 지정해야 하나

같은 글자라도 인코딩에 따라 다른 바이트가 나온다. 
-> 특히 한글의 경우
- "가"를 UTF-8로 -> 3바이트
- "가"를 UTF-16으로 -> 2바이트
저장할 때와 읽을 때 인코딩이 다르면 글자가 꺠진다. 그래서 DB는 인코딩을 하나로 고정(보통 UTF-8)한다.

왜 문자열 "UTF-8" 대신 StandardCharsets.UTF_8?
text.toByteArray(Charset.forName("UTF-8"))   // 문자열로 지정 — 오타 위험 + 예외 가능
text.toByteArray(StandardCharsets.UTF_8)     // 상수로 지정

"UTF-8"처럼 문자열로 쓰면 오타("UTF8","utf-9")를 컴파일러가 못 잡고 런타임에 터진다.
StandardCharsets.UTF_8은 미리 정의된 상수라 오타가 원천 불가능하며, 매번 이름으로 찾는 비용도 없다.

주요 상수: UTF_8, UTF_16, US_ASCII, ISO_8859_1

둘이 함께 쓰이는 실제 패턴 (문자열 직렬화)

val name = "가나다"
val nameBytes = name.toByteArray(StandardCharsets.UTF_8) // 규칙으로 바이트화

val buffer = ByteBuffer.allocate(4 + nameBytes.size)
buffer.putInt(nameBytes.size)   // 먼저 길이를 4바이트로 기록
buffer.put(nameBytes)       // 그 다음 실제 바이트로

여기서 길이를 먼저 쓰는게 핵심이다. 문자열은 길이가 제각각(한글이면 글자당 3바이트)이라, 나중에 읽을 때 "몇 바이트를 읽어야 이 문자열이 끝나는 지" 모른다.
그래서 [길이][실제 데이터] 순서로 저장하면, 읽을 때 먼저 길이(4byte)를 읽고 -> 그 만큼 정확히 잘라 읽을 수 있다.
이걸 length-prefixed encoding이라 부른다.

실제 Type.kt 객체 내 decode를 보면
String decode 3단계를 보자.
```java
STRING -> {
    val len = buffer.int
    val bytes = ByteArray(len)
    buffer.get(bytes)
    String(bytes, StandardCharsets.UTF_8)
}
```
이처럼 인코딩 때 "가나다"(UTF-8로 9바이트)를 저장했다면 버퍼엔 [00 00 00 09][가나다 9바이트] 이렇게 들어있다.

1. val len = buffer.int
   - 커서 위치에서 4바이트를 읽어 9를 얻는다. 그리고 커서를 자동으로 4칸 전진한다.

2. ByteArray(len)
   - len(=9) 크기의 빈 바이트 배열을 만든다. "이만큼 담을 통을 준비"
3. buffer.get(bytes)
   - get에 배열을 넘기면, 그 배열 크기만큼(=9바이트) 버퍼에서 읽어 채운다. 즉 배열 길이가 몇 바이트 읽을 지를 결정한다. 그리고 커서가 또 9칸 전진해서 다음 값 시작점에 멈춘다.
4. String(bytes, UTF_8)
   - 채워진 9바이트를 UTF-8 규칙으로 해석해 "가나다" 복원한다.

buffer.int가 왜 ()가 없나?
java 원본은 buffer.getInt()인데 kotlin에서 buffer.int로 쓰여있다. 이건 kotlin의 프로퍼티 문법이다. 
kotlin이 자바 getxxx() 메서드를 xxx프로퍼티처럼 접근하게 해줘서 그렇다.
```java
buffer.int   ==  buffer.getInt()    // 완전히 같음, 4바이트 읽고 커서 전진
buffer.long  ==  buffer.getLong()   // 8바이트
```

주의 : 이름이 int라는 값을 그냥 "보기만" 하는 것 같지만, 읽으면서 커서를 전진시키는 엄연한 동작이다.
val a = buffer.int; val b = buffer.int를 하면 서로 다른 4바이트를 읽는다.

인코딩 ↔ 디코딩 대칭이 핵심

이 구조가 잘되는 이유는 쓴 순서 = 읽는 순서 
```java
encode:  putInt(길이) → put(데이터)
decode:  int(길이)     → get(데이터)
         ↑ 같은 순서, 같은 크기로 커서가
```

INT/BIGINT는 크기가 고정 (4/8바이트)이라 길이가 필요 없지만, STRING은 가변이라 길이를 먼저 읽어야 데이터를 몇 바이트 잘라낼 지 알 수 있다.
그래서 STRING만 [길이][데이터] 2단계인 것이다.

한 가지 짚어둘 점 — 순서 의존성

val len = buffer.int
val bytes = ByteArray(len); buffer.get(bytes)
